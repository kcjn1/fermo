# Fermo Technical Spike

## Current Decision

Proceed with the repo, model, tests, and dogfood shell now. Do not claim beta readiness until the signed macOS spike also passes the browser matrix and lifecycle checks.

## Spike 1: Website Blocking

Goal: prove Fermo can block selected domains across common browsers.

Implementation path:

- Add an Xcode Network Extension content-filter target for macOS.
- Feed the extension a minimal rule snapshot generated by `FermoCore`.
- Block `reddit.com` and `youtube.com` as test domains.
- Verify Safari, Chrome, Firefox, private windows, sleep/wake, Wi-Fi changes, and reboot.

Pass condition:

- Blocked domains fail closed during an active session.
- Rules clear after session end.
- Permission-denied state is visible in the app.

## Spike 2: App Blocking

Goal: interrupt selected apps during active sessions.

Implementation path:

- Start with `NSWorkspace.shared.runningApplications`.
- Match by bundle identifier.
- Attempt graceful termination first.
- Record apps that relaunch, resist termination, or need stronger handling.

Pass condition:

- A selected app cannot remain usable during an active session under normal user behavior.
- The app explains the limitation honestly.

Current local status:

- `FermoSystem.AppInterruptionController` detects running apps and records interruption outcomes.
- Focus Room interruption now derives policy-violating app targets from the active allowlist path, while excluding Fermo and critical macOS shell apps.
- The dogfood app has `Start App Spike` / `Stop Apps` and `FERMO_AUTOSTART_APP_SPIKE=1`.
- Diagnostic spike policies use Soft rigor so developers can always stop them during local signing and permission work. Real Locked contracts still use `LockedModeGuard` and break-glass.
- Sandboxed app interruption failed on a signed build because macOS denied Apple Events and process signaling.
- Non-sandboxed containing app interruption passed on a signed build with Calculator already running and Calculator relaunched during the active session.
- Website blocking still passed after the containing app sandbox was disabled; the Network Extension target remains sandboxed.
- Detailed handoff: `docs/macos-app-interruption-spike.md`.

## Spike 3: Helper Persistence

Goal: active sessions survive main-app quit and routine Mac lifecycle events.

Implementation path:

- Register a helper using `SMAppService`.
- Store active session state in a shared app group or agreed local container.
- Restore blocking state after main-app relaunch, sleep/wake, and reboot.

Pass condition:

- A locked active session is restored without relying on the main app window staying open.

Current local implementation status:

- `FermoHelper` now exists as a real Xcode macOS app target, not only a SwiftPM executable.
- `Fermo.app` embeds `FermoHelper.app` into `Contents/Library/LoginItems`.
- `FermoSystem.HelperRegistrar` wraps `SMAppService.loginItem(identifier:)` with testable status, register, and unregister seams.
- The dogfood app has `Start Helper Spike`, `Unregister`, and `Open Login Items`.
- The helper diagnostic spike remains explicitly reversible; use the real Start Contract flow when validating Locked contract break-glass behavior.
- The helper reads `FermoSnapshot.json` from the shared app group, refreshes the content-filter rule snapshot, and interrupts blocked apps while a persisted session is active.
- Runtime stop/cleanup now goes through `FermoSystem.ProtectionRuntimeController`, so diagnostic stop flows, terminal evidence cleanup, shared snapshot persistence, and Network Extension deactivation are covered by package tests instead of living only in the SwiftUI view model.
- Signed runtime validation on 2026-05-17 passed the main-app-quit gate: `FermoHelper` stayed running, interrupted Calculator, refreshed shared rule snapshots, and reddit/youtube stayed blocked while `example.com` loaded.
- Sleep/wake and reboot/login restore remain manual follow-up checks.
- Detailed handoff: `docs/macos-helper-persistence-spike.md`.

## Known Constraints

- Network Extension behavior depends on entitlements, signing, and user approval.
- Endpoint Security may be needed for stronger process enforcement, but it increases review, permission, and distribution complexity.
- Swift Package code in this repo models the integration boundaries; the real extension target needs Xcode packaging.
- Xcode 26 exposes the macOS `filter-data` provider as a System Extension target, embedded in the containing app.

## Spike 1 Implementation Handoff

- Xcode project: `Fermo.xcodeproj`.
- App target: `Fermo`.
- Filter target: `FermoFilterExtension`.
- Provider class: `FermoFilterDataProvider`.
- Snapshot model: `FermoCore.ContentFilterRuleSnapshot`.
- Dogfood trigger: `Start Website Spike` / `Stop` in the menu bar popover and main window.
- Signing and approval checklist: `docs/macos-network-extension-signing.md`.
- The website spike is intentionally a stoppable diagnostic policy. Locked contract semantics are validated through Start Contract, not by trapping the developer in a spike.

Current local status:

- Swift Package build and tests pass.
- The Xcode app and embedded system extension build with Apple Development signing.
- The local system extension reaches `activated enabled`.
- The `Fermo Focus Filter` Network Extension session reaches `connected`.
- `curl`, Safari, and Chrome trigger provider drops for `www.reddit.com` and `www.youtube.com`.
- The provider applies `NEFilterSettings(defaultAction: .filterData)` at startup so socket flows are handed to the data provider, then reads `FermoCore.ContentFilterRuleSnapshot` from the shared app group.
- The provider now reloads the shared app-group rule snapshot when it changes, so helper-refreshes take effect without relying on a stale in-memory snapshot.
- Signed runtime validation on build `0.1.0/3` passed after main-app quit: `FermoHelper` stayed running, interrupted Calculator, and reddit/youtube stayed blocked while `example.com` loaded.
- Replacing a system extension build requires a higher `CFBundleVersion`; the local `0.1.0/3` build replaced `0.1.0/2` and `0.1.0/1`, which remain `terminated waiting to uninstall on reboot`.
- Firefox is not installed on the local development machine, so Firefox validation remains unchecked.
